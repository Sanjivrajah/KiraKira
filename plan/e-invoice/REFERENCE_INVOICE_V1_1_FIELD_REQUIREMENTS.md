# Reference: MyInvois Invoice v1.1 Field Requirements

## Status and authority

Verified online on **17 July 2026** against:

- IRBM e-Invoice Guideline Appendix 1, which defines the 55 business data fields.
- IRBM e-Invoice Guideline Appendix 2, which defines additional annexure fields.
- MyInvois SDK Invoice v1.1, which defines the implementation-level UBL 2.1 structure, cardinality, character limits, and conditional requirements.

Official sources:

- https://www.hasil.gov.my/en/e-invoice/reference-for-the-implementation-of-e-invoice/guidelines/
- https://www.hasil.gov.my/media/fzagbaj2/irbm-e-invoice-guideline.pdf
- https://sdk.myinvois.hasil.gov.my/documents/invoice-v1-1/

The HASiL guidelines page displayed **e-Invoice Guideline Version 4.7** on the verification date, but its linked PDF identified itself internally as **Version 4.6, published 7 December 2025**. Because of that publication mismatch, Codex must re-check the guidelines page, linked PDF version, SDK Invoice v1.1 page, release notes, and code lists before implementing production rules. Record the verification date and source versions in code or fixtures.

For API payload shape, the current MyInvois SDK Invoice v1.1 page is the technical implementation contract. The 55-field appendix is a product and data-capture checklist; it is not by itself a complete JSON/XML schema.

## Requirement legend

- **M — Mandatory:** Always required for a standard Invoice v1.1, subject only to an explicitly documented scenario substitution such as consolidated/general-public values.
- **C — Conditional:** Becomes mandatory when its stated condition applies.
- **O — Optional:** Must not block readiness when absent.
- **Derived:** Mandatory or optional output calculated or generated by the system; it should not normally be a manual field.

The preparation UI must evaluate applicability before declaring a record ready. It must never ask every user to enter every optional field.

### Core and tax structure

- Date and time are separate UBL elements.
- Billing period contains separate optional start and end dates.
- Invoice currency is mandatory; tax currency is optional.
- A non-MYR invoice requires the source currency, MYR target currency, and calculation rate mapping.
- Total Tax Amount is mandatory.
- Total Tax Amount Per Tax Type is mandatory with one or more tax subtotals in the SDK.
- Tax exemption requires the exemption reason, zero/exempt tax treatment, and exempt taxable amount mappings.
- Invoice-level discounts and charges are repeatable allowance/charge structures, not single scalar fields.
- Line classification is repeatable; at least one `CLASS` code is mandatory.
- Prepayment may include optional time in addition to the guideline's amount, date, and reference fields.
- Signature is a multi-element UBL extension, not one string field.

## Scenario overlays

The tables above are the standard Invoice v1.1 baseline. Before implementation, create explicit overlays for:

- Standard Malaysian B2B/B2G invoice.
- Consolidated e-Invoice / General Public.
- Foreign buyer.
- Self-billed invoice.
- Credit note, debit note, and refund note.
- Tax-exempt transaction.
- Foreign-currency invoice.
- Import/export of goods.
- Shipping recipient different from buyer.

An overlay may prescribe placeholder TIN, registration, address, contact, state, or classification values. Such values must come from current official guidance; Codex must not invent them or generalise a value from one scenario to another.

## Known Niaga gaps to verify in Stage 1

- Live Supabase business loading currently does not return persisted supplier TIN and registration details reliably.
- Live invoice persistence reduces supplier and buyer snapshots and loses some rich canonical fields.
- Line classification, unit, tax type, rate, discounts, and charges are partly persisted, but the assembled canonical document must be round-trip tested.
- Optional billing, prepayment, shipping, trade, customs, tariff, and origin fields are not yet a complete persistence/UI flow.
- UBL Invoice v1.1 mapping and signature generation are not yet implemented.
- Existing readiness rules cover only a subset of this verified matrix and need a coverage test.
